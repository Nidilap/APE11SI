RESPOSTAS AS PERGUNTAS

  • Para que serve o commit e qual a melhor forma de utilizá-lo?

    • O commit serve para salvar as mudanças feitas no repositório local em uma nova versão.
    Eles salvam a versão do código e deixam guardada de forma segura.
    Após salvar essa versão, o código ainda não é mandado para o servidor, sendo necessário executar o "push" para enviá-las.
    É recomendado que a cada nova mudança seja feito um commit, visto que salvam os códigos de forma segura.
    Antes de realizar o commit, é necessário adicionar os arquivos que seram "commitados" através do comando git add.
    Com isso, é possível "commitar" apenas as mudanças que deseja ou separar arquivos alterados em diversos commits de acordo com a necessidade.

  • Qual a diferença entre commit e push?

    • Enquanto que o commit serve para criar uma nova versão local do projeto, o push serve para enviar essas mudanças para o repositório central.
    Ao dar o push, muitas vezes é possível dar conflito, sendo necessário realizar um pull antes de fazer o push, para resolver possíveis conflitos.
    O push é feito usando o comando "git push".
    Pode receber alguns parâmetros, sendo o principal deles o repositório, que pode ser apenas origin, mandando para o repositório padrão.
    Além deste, o push também permite alguns parâmetros opcionais caso for necessário, como o --push-option, que permite várias opções, como o --force, que força o push mesmo que ocorra erro.

  • O que é tag?

    • As tags são usadas para marcar pontos específicos do projeto como sendo importantes.
    Geralmente são usadas para lançamento de novas versões do software (chamadas de releases).
    No Git, são divididas em dois tipos: Leve e Anotada.
    A tag leve é muito parecida com uma branch que não muda, apontando basicamente para um commit específico.
    A anotada, por outro lado, são mais complexas e servem para armazenar completamente os objetos no banco de dados do Git.
    Possuem checksum, marcações de nome, email e data, além de ter detalhes sobre a tag em si.
    São recomendadas quando se tem todas as informações necessárias para a release.

  • O que é branch?

    • Vem do inglês e significa ramificação. COmo o nome diz, é basicamente uma ramificação do projeto, divergindo da linha principal de desenvolvimento sem alterá-la.
    Em muita ferramentas de versionamento sem branches, é necessário criar uma nova cópia do repositório para fazer essas modificações.
    No git, o sistema de ramificação é facilitado e é tido como seu "recurso matador", que mais diferencia de outros sistemas de controle de versão.
    A branch é especialmente útil para tarefas "epics", tarefas muito grandes que demandam um tempo maior para desenvolvimento mas que não devem atrapalhar o fluxo normal do projeto (resolução de bugs e outros problemas).
    Pode ser útil em alterações da base da estrutura ou algumas mudanças muito grandes na API do projeto, por exemplo.

  • Como resolver possíveis conflitos no Git?

    • Geralmente, ao dar um push das mudanças no repositório principal ou dar um merge de branch, maior parte dos arquivos serão mesclados sem nenhum conflito.
    Porém, caso o mesmo arquivo tenha sido alterado, pode ocorrer o que é chamado de "merge conflict", ou conflito de mescla. Quando isso ocorre, é necessário que você defina o que será mantido e o que será descartado (se o que está no repositório central é o certo ou se o que você fez é o certo). Muitas vezes, não se descarta um ou outro, mas são feitos alguns filtros para manter o que é necessário das duas mudanças. Ao ocorrer o conflito, são separados em duas partes dentro do arquivo, HEAD (repositório central) e a branch atual (ou o repositório local).
